# Описание проекта

## Общее назначение
**ML Portal** — локальный корпоративный AI-ассистент для департаментов компании.

Каждый департамент (отдел) работает как отдельный тенант со своими настройками, но имеет доступ к общим ресурсам и данным других департаментов через систему политик доступа. Пользователи взаимодействуют с AI-агентами, которые используют инструменты (RAG, Jira, NetBox и др.) для решения задач.

Проект поддерживает cross-tenant доступ, иерархические права, гибкую настройку агентов и graceful degradation при недоступности инструментов.

## Основной функционал
- **Agent Runtime**: Tool-call loop для выполнения сложных задач через инструменты
- **RAG Pipeline**: Загрузка документов, извлечение текста, чанкинг, мультимодельный эмбеддинг, векторный поиск
- **Permission System**: Гранулярные политики доступа (User > Tenant > Default) к инструментам и коллекциям
- **Tool Instances**: Подключения к внешним системам (Jira, NetBox) с credential management
- **Collections**: Динамические таблицы с SQL и векторным поиском
- **Prompt Management**: Версионирование промптов (draft/active/archived), Baseline для ограничений
- **Admin Panel**: Управление пользователями, департаментами, агентами, промптами, инструментами
- **Real-time Updates**: SSE для live-статусов RAG pipeline и чат-стримов
- **Audit Logging**: Отслеживание всех действий пользователей и запусков агентов

## Публичные API
Аутентификация (JWT), разделы **Chats**, **RAG Documents**, **Analyze**; поддержка стрима, пагинации и rate-limit заголовков.  
Примеры ручек:
- `POST /api/auth/login`, `POST /api/auth/register`
- `POST /api/chats`, `GET /api/chats/{id}/messages`, `POST /api/chats/{id}/messages` (**response_stream: true** для SSE)
- `POST /api/rag/upload`, `GET /api/rag/`, `GET /api/rag/{id}`, `POST /api/rag/search`, `PUT /api/rag/{id}/tags`, `POST /api/rag/{id}/archive`, `DELETE /api/rag/{id}`
- `POST /api/analyze/upload`, `GET /api/analyze/`, `GET /api/analyze/{id}/download`
- WebSocket-канал для стриминга чата.

## Архитектура

### Backend
- **FastAPI** — асинхронный API layer
- **PostgreSQL** — основная БД (пользователи, агенты, промпты, политики)
- **Qdrant** — векторное хранилище для RAG и коллекций
- **Redis** — кэш, очереди, rate limiting
- **Celery** — фоновые задачи (RAG pipeline, векторизация коллекций)
- **MinIO/S3** — хранение файлов и артефактов
- **Docker** — контейнеризация (Compose для dev, Kubernetes для prod)

### Frontend
- **React 18 + TypeScript + Vite**
- **TanStack Query** — серверное состояние
- **Zustand** — локальное UI состояние
- **CSS Modules** — стилизация
- **SSE** — real-time обновления

### Key Design Decisions
- **Scope-based isolation**: Default → Tenant → User для политик и кредов
- **Agent Runtime**: Tool-call loop с pre-runtime маршрутизатором (AgentRouter)
- **Credential resolution**: User > Tenant > Default приоритет
- **Versioned prompts**: Draft/Active/Archived для контроля изменений
- **Baseline merge**: Default baseline + Agent baseline с приоритетом агента
- **Graceful degradation**: Partial mode для агентов с недоступными инструментами

## Дополнения к MVP (из опыта)

Чтобы MVP был жизнеспособным и не встал на типичных проблемах, добавлены минимальные требования:

- Управляемость пайплайнов: статусы, отмена, перезапуск, idempotency.
- Дедупликация контента и нормализация (контент-хэш, стабильно формируемые chunk_id).
- Схема метаданных для RAG (tenant, source_id, chunk_id, page, lang, mime, version, updated_at, tags[]).
- Гибридный поиск (векторный + лексический fallback), агрегация результатов (RRF/нормализация).
- Кэширование ответов и инвалидация при обновлении источников.
- Минимальная безопасность: ротация JWT-ключей, строгие границы tenant в DAL, аудит админ-действий.
- Наблюдаемость: healthz/readyz, p95 latency, 5xx, глубина очередей, корреляция request_id.
- Nginx-настройки для SSE/WS (proxy_buffering off, таймауты, лимиты upload).
- Лимиты и backpressure для LLM/EMB, dead-letter очереди для Celery.
- UX-мелочи: плашка maintenance, предсказуемые ответы на пустую выдачу, версия билда в UI.

Пункты CI/CD smoke-тестов и автоматических бэкапов отмечены как **заглушки** — закладываются места в коде и структуре, но реализуются после основных фич.

## Backend Structure

```
apps/api/src/app/
├── core/              — Configuration, middleware, errors
├── models/            — SQLAlchemy models
├── schemas/           — Pydantic schemas
├── repositories/      — Data access layer
├── services/          — Business logic
├── api/               — FastAPI routers
│   ├── v1/            — API v1 endpoints
│   └── mcp/           — MCP protocol
├── agents/            — Agent Runtime
│   ├── runtime.py     — Tool-call loop
│   ├── router.py      — Pre-runtime router
│   ├── handlers/      — Tool handlers
│   └── builtins/      — Built-in tools
├── workers/           — Celery tasks
└── adapters/          — External clients
```
