# Embedding Model Management

## Цель

Управление жизненным циклом embedding/rerank моделей находится в зоне ответственности админа платформы.
Тенанты управляют только документами в своих коллекциях.

## Роли и границы

- Админ:
  - добавляет/обновляет/удаляет модели;
  - запускает миграции и массовый реиндекс;
  - контролирует фоновые задачи и их статусы.
- Тенант:
  - загружает/удаляет документы в коллекциях;
  - не запускает массовые межмодельные операции.

## Состояния модели

- `active`: модель доступна и может использоваться для поиска/индексации.
- `inactive` (контейнер/endpoint недоступен): модель есть в БД, но недоступна в рантайме.
- `deleted`: модель удалена админом.

## Состояния эмбеддингов документов

- `ready`: векторы доступны и используются.
- `archived`: векторы сохранены, но не используются для retrieval (например, модель недоступна/удалена).
- `stale`: векторы требуют реиндекса на целевую модель.

## Политика автоматических действий

1. Модель отключилась (`active -> inactive`):
- фоновая задача переводит связанные векторы в `archived`.
- retrieval не использует archived-векторы.

2. Модель восстановилась (`inactive -> active`):
- фоновая задача снимает архивный режим (unarchive) там, где это безопасно.
- если обнаружена несовместимость/дрейф — помечает как `stale`.

3. Админ удалил модель (`deleted`):
- фоновая задача архивирует связанные векторы;
- модель нельзя использовать в новых операциях.

## Ручные операции админа

1. Добавление новой модели:
- по умолчанию не запускает авто-реиндекс всех документов;
- админ вручную инициирует массовую миграцию/реиндекс.

2. Миграция между моделями (`source -> target`):
- scope: все документы / tenant / коллекции;
- поддержка dry-run (оценка объема), progress, cancel, retry;
- результат отражается в статусах документов и задач.

3. Переключение основной/дефолтной модели:
- допускается после завершения ключевой миграции;
- при нехватке векторов целевой модели retrieval работает по зафиксированной fallback-политике.

## Мультимодельный сценарий

- У документа могут быть векторы для нескольких моделей.
- Runtime выбирает векторы по приоритету:
  1) активная основная модель;
  2) разрешенный fallback (если включен политикой);
  3) иначе degraded/empty retrieval результат.

## UI/Control Plane (админка)

У модели должна быть вкладка управления:

- `Использование`:
  - где модель выбрана (tenant defaults/overrides, коллекции, число документов/векторов).

- `Миграции`:
  - запуск `source -> target`;
  - dry-run;
  - прогресс/статус задач;
  - журнал ошибок и ретраев.

## Ограничения безопасности и удаления

- Удаление модели:
  - мягкое (логическое) удаление допустимо сразу;
  - физическое удаление векторов — только после подтверждения объема и последствий.

- Запрет разрушения без подтверждения:
  - каскадное удаление векторов требует явного confirm.

## Наблюдаемость

- Метрики:
  - число `archived/stale/ready` векторов по моделям;
  - длительность и throughput миграций;
  - доля ошибок по стадиям.

- Аудит:
  - кто запустил миграцию;
  - с каким scope;
  - итог (успех/частичный/ошибка).

## Принцип

Автоматизируем только безопасные реактивные действия (archive/unarchive по состоянию модели).
Тяжелые и дорогостоящие операции (массовый реиндекс/межмодельная миграция) выполняются вручную админом.
