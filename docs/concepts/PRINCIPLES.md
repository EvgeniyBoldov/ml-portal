# Ключевые принципы

## Архитектурные принципы

### 1. Scope-based Isolation
Иерархия прав и настроек: **Default → Tenant → User**.
- Default — базовые настройки для всех
- Tenant — переопределения для департамента
- User — индивидуальные настройки пользователя

Приоритет при резолве: **User > Tenant > Default**.

### 2. Tool-call Loop
Агенты работают через цикл вызова инструментов:
1. LLM генерирует намерение
2. Router определяет доступные инструменты
3. Runtime выполняет tool calls
4. Результаты возвращаются в LLM
5. Цикл повторяется до завершения

### 3. Graceful Degradation
Если инструмент недоступен:
- `supports_partial_mode=true` — агент продолжает с warning
- `supports_partial_mode=false` — агент недоступен

### 4. Versioned Prompts
Промпты имеют версии со статусами:
- **draft** — черновик, не используется
- **active** — активная версия
- **archived** — архивная, только для истории

### 5. Baseline Merge
Системные ограничения через baseline промпты:
- Default baseline — глобальные ограничения
- Agent baseline — специфичные для агента
- Merge: `default + agent` с приоритетом агента

## Продуктовые принципы

### 1. Оффлайн-first
Все модели и данные хранятся локально. Никаких внешних запросов в проде.

### 2. Tenant Isolation
Данные департаментов изолированы. Cross-tenant доступ только через явные политики.

### 3. Audit Everything
Все действия логируются: API запросы, изменения конфигурации, запуски агентов.

### 4. Configuration over Code
Поведение системы настраивается через конфигурацию, а не код:
- Агенты — через промпты и политики
- Права — через PermissionSet
- Инструменты — через ToolInstance

## Технические принципы

### Backend
- Асинхронность везде (async/await)
- Полная типизация (type hints)
- Repository pattern для data access
- Service layer для бизнес-логики
- flush() в репозиториях, commit() в роутерах

### Frontend
- Domain-first структура
- TanStack Query для серверного состояния
- Zustand для UI состояния
- CSS Modules для стилей
- Компоненты < 250 строк
