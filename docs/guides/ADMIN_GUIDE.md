# Admin Guide

Гид для platform admin и tenant admin: как устроена система управления, что на что влияет, и где типичные риски.

Сопутствующие практические инструкции:

## 1. Зона ответственности администратора

Администратор управляет:

- пользователями и ролями,
- tenant-ами и их настройками,
- агентами, политиками, промптами,
- инструментами и инстансами,
- доступами (permissions) и credential sets,
- эксплуатационным состоянием (runs, health, audit).

## 2. Основные сущности и связи

- **Agent**: исполняемый профиль поведения.
- **Prompt/Baseline**: инструкции и рамки.
- **ExecutionLimits**: единая модель лимитов исполнения (platform / orchestrator_role / agent).
- **Tool + ToolInstance**: функциональность и конкретное подключение.
- **PermissionSet**: кто что может использовать.
- **CredentialSet**: какими кредами и на каком scope подключаемся.
- **Collection/RAG**: корпоративные данные для поиска и контекста.

## 3. Scope и приоритеты

Критичный инвариант: разрешение идет по цепочке:

1. user
2. tenant
3. default

Приоритет всегда: **User > Tenant > Default**.

Это влияет на:

- доступы к инстансам/инструментам,
- выбор кредов,
- итоговый runtime-контур пользователя.

## 4. Роли и ограничения

## 4.1 Platform Admin

Может:

- управлять всей платформой,
- менять default настройки,
- управлять системными сущностями глобально.

Риски:

- изменение default scope может затронуть всех пользователей.

## 4.2 Tenant Admin

Может:

- управлять данными и доступами внутри своего tenant-а,
- назначать tenant-level правила и креды.

Не может:

- менять глобальные системные настройки,
- управлять чужими tenant-ами.

## 5. Где что настраивается в админке

- `Настройки платформы -> Лимиты`
  - только platform-level execution limits (LLM + Runtime).
- `Оркестрация -> Planner / Synthesizer / Fact Extractor / Summary Compactor`
  - у каждой системной роли свои execution limits (LLM + Runtime) и свой prompt contract.
- `Агенты -> <агент> -> Лимиты`
  - execution limits конкретного агента (LLM + Runtime).
- `Настройки платформы -> Политики/гейты`
  - policy gates (`forbid_*`, `require_confirmation_*`) и policy text.

Важно:
- лимиты исполнения больше не ведутся через platform caps;
- единый источник лимитов — `execution_limits`.

## 6. Операционный минимум (ежедневный контроль)

1. Проверить failed agent runs.
2. Проверить health tool instances.
3. Проверить ошибки ingestion/RAG pipeline.
4. Проверить изменения прав/кредов в audit.

Если где-то много красного — сначала стабилизируем, потом улучшаем. Не наоборот.

## 7. Типовые административные сценарии

## 7.1 Выдать доступ пользователю к инструменту

1. Проверить роль и tenant пользователя.
2. Обновить PermissionSet на нужном scope.
3. Проверить effective permissions.
4. Прогнать тестовый запуск агента.

## 7.2 Подключить новый tool instance

1. Создать/обновить ToolInstance.
2. Добавить credential set на нужном scope.
3. Проверить health status.
4. Привязать instance к агенту (binding) и проверить policy limits.

## 7.3 Включить новый источник данных

1. Создать/настроить collection или RAG ingestion.
2. Убедиться в корректности schema/metadata.
3. Проверить поиск и фильтрацию по tenant/scope.
4. Открыть доступ через PermissionSet.

## 8. Lifecycle удаление (soft/hard)

- Любая сущность lifecycle сначала может быть переведена в `deprecated` (soft delete).
- Hard delete доступен отдельно, опционально с каскадным удалением зависимостей.
- Перед удалением нужно смотреть граф зависимостей (что удалится, что перепривяжется).
- Физическое удаление deprecated-сущностей выполняется фоновым GC по retention TTL.

## 9. Правила безопасных изменений

- Не менять несколько контуров сразу без проверки.
- Любые изменения прав/кредов — с проверкой effective access policy и scope-правил (tenant = отдел, не жёсткая SaaS-граница).
- Изменения политик — только с оценкой влияния на SLA/стоимость.
- Все критичные изменения должны быть обратимыми.

## 10. Эскалация и взаимодействие

Передавайте в AI engineering, если:

- нужен новый tool handler,
- требуется изменение runtime flow,
- есть системная деградация качества/latency,
- есть повторяющиеся ошибки в ingestion/tool execution.

Админка — это пульт управления, а не генератор сюрпризов. Лучше маленький контролируемый шаг, чем большой героизм в пятницу вечером.
