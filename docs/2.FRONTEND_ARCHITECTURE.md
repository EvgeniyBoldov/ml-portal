# ML Portal — фронтенд-архитектура и потоковые сценарии

> **ВАЖНО**: Этот документ описывает техническую реализацию фронтенда. Для общего понимания архитектуры см.:
> - `PRODUCT_VISION.md` — видение продукта
> - `DATA_MODEL.md` — модели данных
> - `FRONTEND_ARCHITECTURE.md` — полная архитектура фронтенда
> - `ADMIN_STRUCTURE.md` — структура админки
> - `architecture/AGENT_RUNTIME.md` — runtime и backend-потоки агентов
> - `architecture/FLOWS.md` — backend/интеграционные сценарии
> - `BACKEND_REVIEW.md` — текущее техническое состояние backend и план доработок

> Документ отражает текущее "как есть" состояние фронтенда и целевую модель после перехода на единый глобальный стрим, короткие чатовые ответы и контекстные стримы документов.

## 0. Обзор

- **Фронтенд**: React 18 + TypeScript + Vite, React Router, TanStack Query, Zustand, CSS‑modules, SSE.
- **Назначение**: чат с LLM (короткие стримы), панель RAG‑документов с live-статусами, админка.
- **Ключевые фичи**: мультитенантность, разделение scope (local/global), гранулярные стримы (тенант → документ → пайплайн), единый HTTP‑клиент с куками.
- **Цель на релиз**: единый глобальный SSE‑менеджер, детерминированные query key’и с tenantId, handshake для документных стримов, короткие чатовые стримы без WebSocket‑зоопарка.

## 1. Архитектурные принципы

1. **Domain-first структура** (`apps/web/src`):
   - `app/` — провайдеры, маршрутизатор, layout’ы, глобальные стойкие штуки.
   - `domains/<area>/` — страницы, виджеты, хукки (по одной крупной фиче на директорию).
   - `shared/` — переиспользуемые API, ui-примитивы, libs.
2. **Одна точка входа**: `main.tsx` → `<AppProviders>` → `<AppRouter>`.
3. **Стримы**: управляются централизованным провайдером, а не "камерами наблюдения" по страницам.
4. **Асинхронность**: все запросы через общий fetch-клиент (`shared/api/http.ts`), никакого axios-зоопарка.
5. **Серверное состояние** — только React Query; **локальный UI** — только Zustand. Никакого кэша от руки ради "так быстрее".
6. **Нет легаси**: один компонент/хук — один файл, CSS только в `.module.css`, никаких styled-components.

## 2. Провайдеры приложения

```
<AppProviders>
  <ErrorBoundary>
    <QueryClientProvider>
      <AuthProvider>
        <SSEManagerProvider>
          <ToastProvider>
            <GlobalConfirmDialog />
            {children}
```

- **ErrorBoundary**: верхний уровень.
- **QueryClientProvider**: единый `queryClient` с настройками по умолчанию (staleTime 30s, gc 5m, retry 1).
- **AuthProvider**: гидрирует пользователя через `useAuthStore` (Zustand) и не блокирует UI.
- **SSEManagerProvider** (новое имя для текущего `SSEProvider`): один EventSource на всё приложение, управляет подписками и счётчиками клиентов.
- **ToastProvider** / **GlobalConfirmDialog** — UI‑утилиты.

> Прежнее размещение SSE на маршруте `/gpt/rag` устарело. Провайдер переносится в `AppProviders`, чтобы стримы чатов/документов/админки работали синхронно.

## 3. HTTP‑клиент и аутентификация

- **Файл**: `shared/api/http.ts`
- Куки httpOnly, `access_token` в памяти, refresh по бековской ручке `/auth/refresh`.
- Каждому запросу можно напихать `timeout` и `idempotent: true` (добавит `Idempotency-Key`).
- **TODO**: убрать безусловный `console.log('[apiRequest]')` из прода; оставить логирование за флагом `VITE_DEBUG_API`.
- SSE использует `EventSource(url, { withCredentials: true })`, никаких token в query.

## 4. Состояние и Query keys

### 4.1 TanStack Query
- Все запросы через `shared/api` + `shared/api/hooks`.
- **Фабрика ключей** (`shared/api/keys.ts`):
  ```ts
  qk.rag.list({ tenantId, filters })
  qk.rag.detail({ tenantId, docId })
  qk.rag.statusGraph({ tenantId, docId })
  qk.chats.messages({ chatId })
  ```
- `tenantId` обязателен в ключах, даже если бек сам его знает → фронт защищён от кэш-коллизии между разными сессиями/вкладками.
- **Запрет**: руками писать `['rag','list']`. Только через фабрику (проверяет TypeScript).

### 4.2 Zustand
- `entities/auth/model/auth.store.ts`: user + гидрация с refresh.
- `app/store/useChatStore.ts`: только UI (sidebar, выбранный чат, поисковая строка). Нет серверных данных внутри!
- `app/store/app.store.ts`: модалки, фильтры, выбор чекбоксов, RAG view state.
- **Инвариант**: если данные приходят от бекенда → Query, если это переключатель/плашка UI → Zustand.

## 5. Потоковые сценарии

### 5.1 Глобальный EventSource
- Провайдер открывает один EventSource (`config.sseBaseUrl`).
- События классифицируются по `channel` и `type`.
- Менеджер ведёт счётчики подписок в памяти и подтверждает бекенду "нужен канал".

### 5.2 Короткие чатовые стримы
- POST `/chats/{chatId}/messages` → сервер возвращает `stream_id`.
- SSE‑менеджер подписывается на `channel=chat:{chatId}:{streamId}`.
- Стрим длится пока идёт ответ (`delta`) и закрывается по `done`.
- При закрытии вкладки фронт вызывает `DELETE /chats/{chatId}/stream/{streamId}` (best effort).

### 5.3 RAG‑уровни
1. **Tenant feed** — когда открыт `/gpt/rag`: подписка `channel=rag:tenant:{tenantId}`. События: создание документа, изменение scope, теги, аггрегированный статус.
2. **Document view** — когда открыт детальный вид/sidepanel документа: фронт делает `POST /rag/{docId}/stream`. Бэк создаёт очередь `rag:doc:{docId}` и начинает публиковать.
3. **Pipeline/statusbar** — если раскрыт статус пайплайна: фронт подписывается на `rag:doc:{docId}:pipeline` (детализированные шаги `extract`, `chunk`, `embed`).

### 5.4 Управление подписками
- Бэк хранит счётчик подписчиков в Redis (`stream:{channel}:subs`).
- Если счётчик падает в 0 → очередь удаляется.
- SSE‑менеджер отправляет heartbeat (`POST /stream/heartbeat`) раз в N секунд, чтобы бэкенд не решил, что подписчик исчез.
- Вариант "без очереди нельзя писать" реализуется на бэке: если нет ключа очереди — событие не публикуется.

## 6. applyRagEvents и обновление кэша

- `app/providers/applyRagEvents.ts` отвечает за merge SSE→Query.
- Нужно обновить использование ключей: `queryClient.setQueriesData(qk.rag.detail({ tenantId, docId }))` вместо `['rag', 'detail']`.
- Идемпотентность: храним `__last_seq` в кэше, игнорируем старые события.
- Структурный шаринг: в map/filter возвращаем новые объекты только для изменённых элементов.
- При событии `rag.deleted` → `queryClient.removeQueries(qk.rag.detail({ tenantId, docId }))`.

## 7. Маршрутизация

- `/login` — гостевой маршрут.
- `/gpt` — MainLayout (чат + документы).
- `/admin` — lazy-модуль за `AdminGuard`.
- Стримы доступны на любой странице, потому что провайдер глобальный.
- При появлении "контекстного роутера" (чаты с файлами) договор: новые контексты получают собственные `channel` в SSE, но менеджер остаётся один.

## 8. UI и UX правила

- Только компоненты из `shared/ui` (Button, Input, Badge, Modal, Popover, ActionsButton и т.д.).
- Все интерактивные элементы проверяем на доступность: `aria-*`, focus-ловушки, `Esc` закрывает модалки.
- Skeleton’ы → `shared/ui/Skeleton` (никаких inline `<div style>`).
- CSS: `.module.css`, классы `dash-case`, без глобальных правок.
- Компоненты < 250 строк; сложные блоки выносим в подпапки `components` / `hooks`.

## 9. Тестирование

- **Юниты**: Vitest + RTL + MSW.
  - Тестировать `applyRagEvents`, SSE‑менеджер, хукки с Query ключами.
- **E2E (Playwright)**: smoke-сценарий "login → upload → ingest → ready → search → logout" + проверка стримовых апдейтов (док должен обновить статус без перезагрузки).
- **API контракты**: MSW‑ручки, проверяющие handshake (`POST /rag/{id}/stream`, `DELETE ...`).
