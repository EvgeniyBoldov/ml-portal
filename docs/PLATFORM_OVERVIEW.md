# ML Portal — Business Logic & Architecture Overview

Этот документ — верхнеуровневая карта системы: что делает платформа, какие роли есть, какие ограничения действуют и как текут данные между Chat, RAG, Collections, Agent Runtime и Admin.

## 1. Что такое ML Portal

ML Portal — мульти-тенантная AI-платформа для корпоративных сценариев:

- пользователь общается с агентом в чате,
- агент использует инструменты и знания (RAG/Collections),
- админ управляет агентами, политиками, доступами и интеграциями,
- система обеспечивает изоляцию данных, трассируемость и предсказуемость выполнения.

Ключевая единица ценности: **управляемый агентный сценарий** (agent + tools + data + policy + permissions).

## 2. Бизнес-домены платформы

### 2.1 Chat

Назначение:

- интерфейс запроса к агенту,
- потоковый ответ (SSE),
- история диалога и шагов выполнения,
- работа с файлами через вложения и generated file outputs.

Ограничения:

- выполнение ограничивается политикой агента (лимиты шагов, времени, tool calls),
- доступ к инструментам и данным определяется через effective permissions.
- в runtime попадают только валидированные chat attachments, а не сырые байты из запроса.

### 2.2 RAG

Назначение:

- загрузка и индексация документов,
- семантический поиск,
- подача релевантных контекстов в агентные ответы.

Ограничения:

- доступ к документам зависит от scope и tenant membership,
- фильтрация результатов делается по RBAC/tenant правилам,
- качество ответа зависит от корректности ingestion и метаданных.

### 2.3 Collections

Назначение:

- структурированные данные tenant-а,
- динамические схемы полей,
- поиск (exact/like/range/vector),
- использование как источника данных для инструментов/агентов.

Текущая рабочая модель:
- локальная табличная коллекция,
- локальная документная коллекция,
- заготовка под удаленный SQL catalog с отдельным resolver path.

Ограничения:

- schema и search modes валидируются,
- vector search только там, где это явно разрешено схемой,
- доступ к коллекциям контролируется permission sets.

### 2.4 Admin

Назначение:

- управление users/tenants,
- управление агентами, промптами, политиками, тулзами и инстансами,
- настройка default/tenant/user уровней доступа и credential resolution,
- операционный контроль (health, runs, audit).

Ограничения:

- роль и scope ограничивают доступ к действиям,
- изменения должны проходить через versioned сущности (где применимо).

## 3. Роли, возможности и ограничения

## 3.1 User

Может:

- запускать доступных агентов,
- работать с доступными данными и инструментами,
- управлять своими credential set (scope=user, если разрешено).

Не может:

- управлять системными сущностями платформы,
- обходить tenant/scope ограничения,
- изменять глобальные политики.

## 3.2 Tenant Admin

Может:

- управлять сущностями своего tenant-а,
- настраивать tenant-level permissions/credentials,
- контролировать данные и доступы внутри своего контура.

Не может:

- менять global/default настройки платформы (если политика не дает),
- управлять данными других tenant-ов.

## 3.3 Platform Admin

Может:

- управлять всем контуром платформы,
- задавать default policies и baseline поведение,
- управлять каталогом агентов/инструментов/моделей,
- контролировать эксплуатационные и security параметры.

Не должен:

- делать исключения «вручную» в обход политики и журналирования.

## 4. Multi-tenant и scope модель

Система использует иерархию разрешения настроек:

1. **User scope**
2. **Tenant scope**
3. **Default scope**

Приоритет разрешения: **User > Tenant > Default**.

Это касается как минимум:

- permissions,
- credentials,
- поведения агента через связки policy/baseline/tool bindings.

## 5. Потоки данных (крупными мазками)

## 5.1 Chat flow

1. User отправляет сообщение в чат.
2. Если есть вложения, сначала валидируются и связываются с chat turn.
3. Платформа определяет agent profile и контекст выполнения.
4. Runtime pipeline собирает execution request (policy, collections, tools, permissions, credentials).
5. Agent Runtime запускает triage → preflight → planner/runtime loop.
6. События выполнения и ответ стримятся пользователю.
7. Результат, метаданные и file outputs фиксируются в истории/run logs.

## 5.2 RAG ingestion flow

1. Документ загружается в tenant-контур.
2. Документ проходит pipeline (extract/chunk/embed/index).
3. В процессе обновляются статусы и метаданные.
4. Готовые чанки доступны для поиска и цитирования в ответах агента.

## 5.3 Tool execution flow

1. LLM инициирует tool_call.
2. Runtime валидирует доступность инструмента и права.
3. Выполняется handler с ToolContext (tenant_id, user_id, chat_id, scopes).
4. Результат возвращается в runtime loop и влияет на финальный ответ.

## 5.4 Pause / Resume flow

1. Runtime может остановиться на `waiting_input` или `waiting_confirmation`.
2. Пауза фиксируется в `agent_run` и `chat_turn` вместе с pause context.
3. Пользователь подтверждает действие или отправляет уточнение.
4. Продолжение сейчас запускается как новый chat turn в том же чате с сохранением истории.
5. Настоящий mid-run checkpoint resume пока не реализован и рассматривается как отдельный этап развития ядра.

## 6. Какие данные где хранятся и зачем

### 6.1 Agent данные

Хранится:

- идентификатор и описание агента,
- ссылки на prompt/baseline/policy,
- bindings к инструментам,
- execution-параметры и logging level.

Зачем:

- управляемый lifecycle агента,
- воспроизводимость поведения,
- контролируемый rollout изменений.

### 6.2 Tool и Tool Instance данные

Хранится:

- tool slug/contract,
- instance config,
- health status,
- credential references.

Зачем:

- разделить «что умеет инструмент» и «как/куда он подключен»,
- обеспечивать безопасную подстановку кредов по scope.

### 6.3 RAG/Collection данные

Хранится:

- документ и ingestion статус,
- chunk метаданные,
- коллекции и их schema,
- индексы/векторные представления.

Зачем:

- контролируемый semantic retrieval,
- предсказуемое качество и проверяемость источников.

## 7. Нефункциональные принципы

- Tenant isolation by default.
- Structured logging и correlation id.
- Версионирование и миграционная дисциплина.
- Минимум хардкода, максимум конфигурации.
- Изменения архитектуры и бизнес-логики обязаны отражаться в документации.
