# RAG Ingest Business Flow

Этот документ описывает бизнес-логику пайплайна обработки документов (RAG ingest) и правила работы со статусами.

## Общее устройство

1. **Загрузка (upload)** — файл безопасно сохранён, статус `completed`.
2. **Ингест** запускает последовательный ствол задач:
   - `extract`
   - `normalize`
   - `chunk`
3. После чанкинга для каждой целевой модели формируется независимая пара задач:
   - `embed.<model>`
   - `index.<model>`
4. Все статусы хранятся в `RAGStatus`. SSE уведомления отдаются после `flush()`, финальное фиксирование изменений — по `commit()` из Celery-задачи.

## Правила статусов

| Этап                 | Источники данных                        | Результат                                                         |
|----------------------|-----------------------------------------|------------------------------------------------------------------|
| `extract`            | MinIO (оригинал)                        | `extracted_text` в MinIO                                          |
| `normalize`          | текст из `extract`                      | нормализованный JSON в MinIO                                      |
| `chunk`              | JSON из `normalize`                     | чанки + метаданные в Postgres                                     |
| `embed.<model>`      | чанки из Postgres                       | эмбеддинги (сложены в Qdrant через следующий этап, статусы в БД)  |
| `index.<model>`      | эмбеддинги (`embed` результат)          | upsert в Qdrant, метрики по индексу                               |

### Значения `StageStatus`

- `pending` — этап ожидает запуска
- `queued` — задача запрошена Celery
- `processing` — выполняется в данный момент
- `completed` — завершён без ошибок
- `failed` — ошибка при выполнении (содержит `error_short`)
- `cancelled` — отменён (в том числе каскадно при падении предыдущих стадий)

### Допустимые переходы

- `pending → queued | processing | failed | cancelled`
- `queued → processing | cancelled`
- `processing → completed | failed | cancelled`
- `completed → queued` (повторный запуск)
- `failed | cancelled → queued` (retry)

## Каскадные правила

1. **Падение последовательного этапа** (`extract`, `normalize`, `chunk`):
   - текущий этап получает `failed`;
   - все последующие последовательные этапы и *все* модельные ветки (`embed.*`, `index.*`) переходят в `cancelled` (или `pending` при retry);
   - артефакты (S3/БД) остаются для отладки, повторный запуск перезапишет их.

2. **Падение `embed.<model>`**:
   - только конкретная модель переходит в `failed`;
   - соответствующий `index.<model>` автоматически получает `cancelled`;
   - другие модели продолжают работу.

3. **Падение `index.<model>`** не тянет каскад — только этот индекс меняет статус.

4. **Остановка этапа пользователем** (`stop_stage`)
   - выбранный этап помечается `cancelled`;
   - используется та же каскадная логика, что и для ошибок.

5. **Retry**:
   - `retry_stage('embed.X')` переводит в `queued` и сбрасывает только `index.X` → `pending`;
   - `retry_stage('index.X')` влияет только на этот индекс;
   - `retry_stage` для любого последовательного этапа переводит downstream этапы в `pending`.

## Оркестрация Celery

```text
extract → normalize → chunk → group( chain(embed.model1 → index.model1), chain(embed.model2 → index.model2), ... )
```

- Для каждой модели `embed_chunks_model` возвращает результат с эмбеддингами, сразу передаваемый в `index_model`.
- Every task:
  1. переводит свой статус в `processing`;
  2. выполняет бизнес-логику;
  3. завершает статус `completed` или вызывает `notify_stage_error` для `failed`;
  4. фиксирует изменения через `commit()`.

## Аггрегированное состояние документа

`calculate_aggregate_status` работает так:
1. Пока pipeline-этапы содержат `pending/queued/processing` — итоговый статус `processing`.
2. Если хотя бы один pipeline-этап `failed` — документ `failed`.
3. Когда pipeline завершён:
   - все модели `index` → `completed` → документ `ready`;
   - часть моделей `completed`, часть `failed/cancelled` → `partial`;
   - нет ни одного `completed` и все `failed/cancelled` → `failed`;
   - наличие `pending/processing` в индексах → `processing`.

## Архивирование

- `archive_document` создаёт служебный узел `archive` со статусом `completed`.
- `get_document_status` возвращает `archived=true`, если узел существует.
- Для исключения модели из поиска планируется отдельный статус `archive.<model>` (пока не реализовано).

## Повторное добавление моделей

- При появлении новой модели её узлы `embed/index` добавляются как `pending`.
- Повторный запуск пайплайна через `retry_stage('chunk')` или `start_ingest` пересоздаёт только новую ветку без затрагивания старых `completed` моделей.

## Ошибки и уведомления

- `notify_stage_error` / `notify_embed_error` помечают соответствующий этап как `failed` и полагаются на каскад.
- SSE-уведомления отправляются после `flush()`; данные фиксируются после `commit()`.
- На фронте статусы приходят через общий стрим и пер-документные SSE (StatusModal).

## Рекомендации по эксплуатации

- При отладке задач проверяйте цепочку статусов через `/rag/status/{document_id}` и логи Celery.
- Для повторного запуска конкретной модели используйте `retry_stage('embed.model')`.
- Для полного рестарта документа — повторно вызвать `start_ingest` (предварительно убедившись, что стволовые этапы снова `pending`).
- Архивирование документа не влияет на ingest-статусы, только скрывает его из общего списка.
