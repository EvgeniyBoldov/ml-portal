необходимо внедрить версионность и наблюдаемость уже сейчас, так как это затрагивает основные таблицы базы данных.

Общая концепция:
- части агента должны быть версионированными
- части агента должны быть наблюдаемыми

Ахитектура компонентов:
- есть две категории компонентов: версионные и не версионные
- в версионированных есть entity и versionEntity.
- в не версионных есть только entity
- в версионных entity содержит общую информацию о компоненте, а versionEntity содержит версию компонента
- к версионным компонентам можно обращаться по id версии или по id компонента и рекомендованной версии
- к полям entity относятся поля которые не влияют на поведение агента и служат описанием компонента для инженера и другие управляемые блоки
- к полям versionEntity относятся поля которые влияют на поведение агента и служат версионированными частями компонента
- версии могут быть в разных статусах: draft, active, inactive. 
    драфт - версия в разработке, active - версия активна, inactive - версия неактивна
- обращение к компоненту происходит через основной блок, который содержит ссылку на версию в качестве указания версии, которую нужно использовать. 
- агент состоит из промта и нескольких других компонентов, при вызове агента вызываются все компоненты с их рекомендованными версиями и результаты объединяются в единый промт для LLM. 
- при вызове агента создается запись в таблице вызовов агента, которая содержит информацию о вызове агента, включая версии компонентов, которые были использованы.

Структура компонента:
- основной блок компонента
    - название
    - описание
    - слаг
    - ссылка на рекомендуемую версию компонента
    - created_at
    - updated_at
    - created_by
    - updated_by
- версия компонента
    - версия
    - статус
    - created_at
    - updated_at
    - created_by
    - updated_by
    - данные компонента 
    - etc

Архитектура агента:
- агент состоит из промта, полиси, бейслайна, инструкций, системного промта, дополнительных данных, бингов. 
- каждый компонент сам возвращает правильную версию компонента исходя из рекомендуемой версии в основном блоке компонента.
- биндинги это связи тулза + инстанса и стратегии. Соответственно каждый компонент вызывает свою версию.

Архитектура тулзов:
- тулз это компонент живущий в двух мирах, это код в python и метаданные как компонент в базе данных.
- метаданные включают в себя описание тулза, типы параметров, примеры использования, etc. и создаются АИ инженером
- код тулза это python функция которая принимает параметры и возвращает результат
- тулз на беке имеет версию и входные и выходные форматы данных
- информация о тулзе загружается в БД при запуске воркера, и должна обновляться при каждом обновлении кода тулза и его версии
- информация о метаданных тулза обновляется инженером.
- связка версии метаданных и версии кода тулза является версией тулза.
- обновление кода не всегда = обновлению версии метаданных тулза, но всегда = обновлению версии тулза
- тулз имеет как и любой другой версионный компонент основные поля и ссылку на версию. 
- тулзы обьединяются в группы типа jira, gitlab, netbox, etc.

Архитектура Агента:
- состоит из основных полей + компоненты
- компоненты это промты, бейслайны, политики(ограницения), биндинги. 
- биндинг это связка тулза + инстанса и стратегии прав
- в версию входят компоненты. 

Архитектура фронтенда:
- не плодим в навбаре лишних сущностей
- для навигации используем хлебные крошки в хедере страницы. 
- добавляем новый непеиспользуемый компонент entityTable. разбиваем вью на несколько спектов и каждый отображаем в отдельном табе. 
- список сущностей в навбаре:
  - промт
  - бейслайн
  - инструменты
  - агенты
  - политики
  - запуски(executions)
  - пользователи
  - тенанты
  - общие настройки
  - логи
  - etc 
- каждая сущность имеет свою страницу со списком, кроме некоторых, такие как общие настройки, логи, etc
- таблица имеет фильтры, сортировку, пагинацию, поиск. 
- клик по айтему в таблице открывает страницу просмотра сущности и в ней уже есть табы.
- страница вью имееет блок кнопок для действий с сущностью, такие как редактировать, удалить, создать новую версию и т.д.
- таб с версиями показывает список версий сущности. при выборе сущности из списка открывается страница просмотра версии с кнопками действий в зависисмости от статуса версии. 
- для агента есть таб запусками(executions), где можно увидеть все запуски агента и фильтровать их по статусу, дате, etc
- если перейти в execution в корне админки, то там будет таблица с запускаим всех агентов. Execution конкретного агента и всех это одна таблица, но только фильтрованная
- у каждого запуска будет своя вью страница с деталями запуска и возможностями повторить запуск с теми же версиями, сравнить с текущей версией и другие операции. 


Принципиальные возможности:
- пользователь запускает агента с рекомендованными версиями сущностей
- админ может запустить агента с любыми версиями сущностей. 
- админ устанавливает рекомендованные версии сущностей для сущностей
- админ может запускать и сравнивать промты между запусками как того что запустили с рекомендованными версиями, так и любые версии между собой как с выполнением агентами, так и просто сравнение итогового промта. 
- разработчики пишут тулзы в коде и проставляют версии. Схемы входных и выходных данных должны подтягиваться из тайпов pydantic.


 Механизм сбора промта агента:
 - если запускает пользователь в чате, то берет рекомендованные версии сущностей, независимости от прав пользователя, админ он или просто пользователь
 - если в админке запускает админ, то берет выбранные версии сущностей или дефолтные версии
 - для пользователя есть прекоммит, который подтягивает рекомендованные версии сущностей и добавляет их в промт перед запуском. Админ или использует этот прекоммит, или может вручную добавить версии сущностей в промт. Это нужно для разных режимов:
   - разбора юзеровского кейса для воспроизводимости
   - эксперименты с разными версиями
   - замер производительности
   - ручной запуск с конкретными версиями
   - сравнение промтов разных версий

Для инстансов тулзов тоже необходимо отображать в биндах какая ерсия использовалась и возможность сгенерировать промт с этой версией включая инстансы и креды(но только их имена, без самих данных).
