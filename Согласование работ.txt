ПРОЦЕДУРА ИЗМЕНЕНИЯ СЕТЕВОЙ КОНФИГУРАЦИИ
КОМПАНИЯ «РОГА И КОПЫТА»

Действующие лица:
- Начальник сетевого департамента — Бык (далее: Бык).
- Инженеры сети — готовят план, выполняют работы.
- Мониторинг / NOC — Зоркий Глаз Иванович (далее: ЗГИ).

----------------------------------------------------------------------
1. ИНИЦИАЦИЯ И ПЛАНИРОВАНИЕ РАБОТ

1.1. Инициатор (инженер) формирует запрос на изменение (RFC), в котором указывает:
- цель работ (что меняем и зачем);
- затрагиваемые объекты (коммутаторы, маршрутизаторы, линк, VLAN, VRF, ACL и т.п.);
- ожидаемый эффект и риски;
- окно проведения (дата/время начала и окончания, часовой пояс);
- план действий (step-by-step, с командами);
- план отката (rollback, также с командами);
- тесты после работ (проверки доступности, pings, трассировки, сервисные проверки).

1.2. План обязательно содержит конкретные команды:
- Cisco:
  conf t
   interface GigabitEthernet0/1
    description UPLINK_TO_CORE
    switchport trunk allowed vlan add 50
  end
  write memory
- Huawei:
  system-view
   interface GigabitEthernet0/0/1
    description UPLINK_TO_CORE
    port trunk allow-pass vlan 50
  quit
  save
- Juniper:
  configure
   set interfaces ge-0/0/1 description "UPLINK_TO_CORE"
   set interfaces ge-0/0/1 unit 0 family ethernet-switching vlan members USERS_OFFICE_2
  commit

----------------------------------------------------------------------
2. СОГЛАСОВАНИЕ ПЛАНА С НАЧАЛЬНИКОМ (БЫК)

2.1. Инженер направляет подготовленный план работ Быку.

2.2. Бык проверяет:
- корректность архитектурных решений;
- соответствие стандартам компании (VLAN, IP-планы, описания интерфейсов);
- риски простоя критичных сервисов;
- наличие рабочего rollback-плана;
- адекватность временного окна.

2.3. При необходимости Бык вносит правки и возвращает план инженеру на доработку.

2.4. После согласования Бык присваивает статус:
- «УТВЕРЖДЕНО» — можно переходить к следующему этапу;
- «ОТКЛОНЕНО» — работы запрещены до переработки плана.

----------------------------------------------------------------------
3. СОГЛАСОВАНИЕ С МОНИТОРИНГОМ (ЗОРКИЙ ГЛАЗ ИВАНОВИЧ)

3.1. Инженер направляет утверждённый Быком план ЗГИ.

3.2. ЗГИ проверяет:
- наличие в плане чётко указанных временных окон;
- список затрагиваемых сервисов и клиентов;
- корректность указания точек мониторинга (интерфейсы, ping, сервисы).

3.3. ЗГИ подготавливает:
- профиль мониторинга на время работ;
- дополнительные алерты/дашборды, если нужно;
- фиксирует базовые значения до начала изменения (baseline).

3.4. ЗГИ подтверждает готовность отметкой «МОНИТОРИНГ ГОТОВ».

----------------------------------------------------------------------
4. ПОДГОТОВКА К ВЫПОЛНЕНИЮ РАБОТ

4.1. Обязательные условия перед стартом:
- есть актуальный бэкап конфигураций затрагиваемых устройств;
- есть доступ к оборудованию по out-of-band (консоль/менеджмент);
- подтверждённое окно от всех заинтересованных сторон.

4.2. Инженер:
- сверяет план действий и отката;
- проверяет свободное место в NVRAM/Flash для сохранения конфигурации;
- подготавливает шаблоны команд в текстовом файле.

----------------------------------------------------------------------
5. ВЫПОЛНЕНИЕ РАБОТ

5.1. В начале окна:
- Инженер уведомляет Быка и ЗГИ о старте.
- ЗГИ включает усиленный мониторинг.

5.2. Инженер выполняет команды строго по шагам плана:
- Пример (Cisco):
  conf t
   interface GigabitEthernet0/10
    description LINK_TO_NEW_ACCESS_SWITCH
    switchport trunk encapsulation dot1q
    switchport mode trunk
    switchport trunk allowed vlan 10,20,30,50
  end
  write memory
- Пример (Juniper ACL):
  configure
   set firewall family inet filter OUTBOUND term ALLOW_USERS from source-address 192.168.10.0/24
   set firewall family inet filter OUTBOUND term ALLOW_USERS then accept
  commit confirmed 10
  # проверка
  commit

5.3. После каждого логического блока:
- Инженер делает промежуточные проверки (ping, traceroute, show интерфейсов);
- ЗГИ подтверждает отсутствие аномалий.

----------------------------------------------------------------------
6. ПРОВЕРОЧНЫЕ ДЕЙСТВИЯ ПОСЛЕ РАБОТ

6.1. Инженер выполняет финальные проверки:
- доступность ключевых сервисов (ping, curl, telnet/ssh к нужным портам);
- состояние интерфейсов:
  - Cisco: show interfaces status
  - Huawei: display interface brief
  - Juniper: show interfaces terse
- проверка маршрутов:
  - show ip route / display ip routing-table / show route

6.2. ЗГИ сверяет показатели с baseline, фиксирует «МОНИТОРИНГ НОРМА».

6.3. При отсутствии проблем:
- Инженер сохраняет конфигурации (write memory / save / commit);
- Фиксируется статус «РАБОТЫ ЗАВЕРШЕНЫ УСПЕШНО».

----------------------------------------------------------------------
7. ОТКАТ (ROLLBACK)

7.1. При критических проблемах:
- Инженер немедленно уведомляет Быка и ЗГИ;
- совещание: откатываемся или продолжает диагностику.

7.2. Откат выполняется по плану:
- Cisco:
  conf t
   default interface GigabitEthernet0/10
  end
  configure replace flash:backup.cfg force
- Juniper:
  rollback 1
  commit

7.3. После отката:
- Инженер сверяет конфиг с бэкапом;
- ЗГИ подтверждает восстановление нормальной работы;
- Статус «ИЗМЕНЕНИЯ ОТМЕНЕНЫ».

----------------------------------------------------------------------
8. ЗАКРЫТИЕ РАБОТ И ДОКУМЕНТИРОВАНИЕ

8.1. Инженер оформляет отчёт:
- что планировалось;
- что выполнено (с перечислением команд, если отличались);
- результаты тестов;
- замеченные проблемы и решения.

8.2. Бык утверждает отчёт или запрашивает доработку.

8.3. ЗГИ обновляет дашборды/алерты с учётом новой конфигурации.

8.4. Документация (схемы, реестры VLAN, IP-планы) обновляется обязательно.
