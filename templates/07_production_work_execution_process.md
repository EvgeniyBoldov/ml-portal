# Регламент проведения работ на продуктивной сети компании "Дубалайн"

## 1. Общие положения

### 1.1. Назначение документа
Настоящий регламент определяет порядок планирования и проведения работ на продуктивной сетевой инфраструктуре компании "Дубалайн". Документ описывает действия инженеров до, во время и после работ.

### 1.2. Область применения
Регламент применяется ко всем изменениям на следующих контурах:
- **PROD-CORE** — магистральная сеть (Core Network)
- **PROD-DC1** — ЦОД Москва (основной)
- **PROD-DC2** — ЦОД Санкт-Петербург (резервный)
- **PROD-OFFICE** — офисная сеть (Campus Network)
- **PROD-DMZ** — демилитаризованная зона
- **PROD-MGMT** — сеть управления (out-of-band)

## 2. Подготовка к работам

### 2.1. Техническая подготовка (за 2+ дня)

#### 2.1.1. Сбор информации
Инженер-исполнитель обязан:
1. Изучить актуальную схему сети и таблицы маршрутизации
2. Проверить текущее состояние оборудования через NOC
3. Убедиться в отсутствии открытых инцидентов на затрагиваемых устройствах
4. Снять резервную копию конфигурации (для всех затрагиваемых устройств)
5. Подготовить команды для выполнения работ
6. Подготовить команды плана отката
7. Убедиться в наличии запасного оборудования (при замене железа)

#### 2.1.2. Проверка доступа
1. Проверить доступность устройств через out-of-band сеть (PROD-MGMT)
2. Проверить наличие консольного доступа (при необходимости)
3. Убедиться в актуальности учетных данных
4. Проверить доступность jump-хостов

#### 2.1.3. Снятие резервных копий
Резервные копии снимаются:
- **Juniper**: `request system snapshot`, `show configuration | save /var/home/backup/pre-change-YYYY-MM-DD.conf`
- **Huawei**: `save`, backup через NMS
- Хранятся в: `\\nas-01.dubaline.local\net-backup\pre-change\`

## 3. Проведение работ

### 3.1. Начало работ

**За 15 минут до начала:**
1. Инженер-исполнитель уведомляет NOC о начале работ: "Начинаю работы по заявке #XXXX"
2. NOC усиливает мониторинг затрагиваемых устройств и сервисов
3. Проверяется актуальное состояние оборудования:
   - Отсутствие активных сигналов тревоги
   - Загрузка CPU и памяти в норме
   - Нет активных инцидентов

**Если обнаружены аномалии:** работы переносятся, инженер уведомляет руководителя.

### 3.2. Пошаговое выполнение

#### 3.2.1. Общие правила
- Работы выполняются строго по согласованному плану
- После каждого шага — проверка состояния оборудования и сервисов
- Один шаг = одно логическое изменение
- Не выполнять несогласованные действия

#### 3.2.2. Порядок выполнения

**Шаг 1. Подтверждение резервных копий**
```
Убедиться, что резервные копии актуальны (не старше 1 часа до начала работ)
```

**Шаг 2. Предварительная проверка**
```
Проверить таблицы маршрутизации, состояние интерфейсов, сессии BGP/OSPF
Записать исходное состояние
```

**Шаг 3. Выполнение изменений**
```
Последовательно применить согласованные команды
```

**Шаг 4. Верификация после каждого изменения**
```
Проверить состояние оборудования
Проверить доступность ключевых ресурсов
Проверить сигналы мониторинга (мониторинг: monitoring.dubaline.local)
```

**Шаг 5. Финальная проверка**
```
Полная проверка работоспособности сервисов
Запрос у NOC: "Нет ли сигналов тревоги?"
Подтверждение у владельца сервиса (при необходимости)
```

### 3.3. Ключевые проверки по контурам

#### PROD-CORE
```
show bgp summary           # Состояние BGP-сессий
show ospf neighbor         # Состояние OSPF-соседей
show mpls lsp              # Состояние MPLS-туннелей
show route summary         # Таблица маршрутизации
ping <GW-адрес>            # Базовая связность
```

#### PROD-DC1 / PROD-DC2
```
show evpn instance         # Состояние EVPN-инстансов
show vxlan interface       # Состояние VXLAN-туннелей
show mac-address-table     # Таблица MAC-адресов
show spanning-tree         # Состояние STP
```

#### PROD-OFFICE
```
show vlan                  # VLAN-таблица
show spanning-tree         # STP
show mac-address-table     # MAC-таблица
show power inline          # Статус PoE (если применимо)
```

### 3.4. Взаимодействие с NOC во время работ

NOC в процессе работ:
- Непрерывно мониторит затрагиваемые устройства и сервисы
- Сообщает исполнителю о любых аномалиях
- Проверяет SLA-метрики через Zabbix / Prometheus (monitoring.dubaline.local)
- Готов инициировать план отката по команде исполнителя

**Контакт с NOC во время работ:**
- Телефон дежурного NOC: +7 (495) 123-45-67 (внутренний: 1001)
- Чат: #noc-duty (корпоративный мессенджер)

## 4. Ситуации и решения в процессе работ

### 4.1. Критерии приостановки работ
Работы немедленно приостанавливаются, если:
- Зафиксировано нарушение доступности сервиса
- Время простоя превысило согласованное
- Оборудование не отвечает на управляющие команды
- NOC фиксирует критические алерты

### 4.2. Процедура отката

**Принятие решения об откате:**
1. Инженер-исполнитель принимает решение об откате
2. Уведомление NOC: "Начинаю откат по заявке #XXXX"
3. Выполнение команд плана отката
4. Проверка восстановления сервисов
5. Уведомление NOC: "Откат завершён, сервисы восстановлены / не восстановлены"

**Кто принимает решение об откате:**
- Инженер-исполнитель — самостоятельно
- NOC — если инженер недоступен или не реагирует
- Руководитель отдела — в случае разногласий

### 4.3. Эскалация
```
Уровень 1: Инженер-исполнитель
Уровень 2: Ведущий инженер направления (Алексей Воронов для Core, Михаил Сергеев для DC)
```

## 5. Завершение работ

### 5.1. Действия по завершению
1. Уведомить NOC: "Работы по заявке #XXXX завершены успешно"
2. Снять резервную копию актуальной конфигурации (post-change)
3. Проверить отсутствие незафиксированных изменений в конфигурации
4. Обновить документацию по схеме сети (при необходимости)
5. Закрыть заявку в системе управления изменениями с отчетом

### 5.2. Проверки после завершения

**Немедленно (в течение 30 минут):**
- Все затронутые сервисы работают штатно
- Нет новых алертов в мониторинге
- Владелец сервиса подтвердил восстановление (для категорий 1-2)

**В течение 24 часов:**
- Анализ логов на предмет ошибок
- Проверка статистики трафика
- Сохранение post-change конфигурации

### 5.3. Отчет о выполнении работ

Обязательно заполняется в системе управления изменениями:

```
Заявка #: XXXX
Дата и время начала: DD.MM.YYYY HH:MM
Дата и время окончания: DD.MM.YYYY HH:MM
Фактический downtime: X мин (или "без прерывания")
Результат: Успешно / Откат выполнен / Откат не выполнен
Описание выполненных работ: ...
Проблемы, обнаруженные в процессе: ...
Действия при откате (если выполнялся): ...
Исполнитель: ФИО
```

## 6. Особые ситуации

### 6.1. Потеря управления оборудованием
1. Переключиться на консольный доступ (PROD-MGMT)
2. Попытаться восстановить конфигурацию
3. При невозможности — уведомить NOC и руководителя
4. При необходимости физического доступа — выехать на площадку

**Адреса площадок:**
- ЦОД Москва (PROD-DC1): г. Москва, ул. Складочная, д. 1 (контакт: datacenter-dc1@dubaline.local)
- ЦОД Санкт-Петербург (PROD-DC2): г. Санкт-Петербург, Пулковское ш., д. 40 (контакт: datacenter-dc2@dubaline.local)

### 6.2. Обнаружение уязвимости в процессе работ
1. Немедленно уведомить инженера по безопасности (Наталья Исаева, network-security@dubaline.local)
2. Не раскрывать информацию об уязвимости
3. Продолжить работы только после оценки рисков инженером по безопасности

### 6.3. Обнаружение несанкционированных изменений
1. Зафиксировать состояние
2. Уведомить инженера по безопасности
3. Приостановить плановые работы
4. Инициировать расследование инцидента
