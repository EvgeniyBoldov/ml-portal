# Регламент согласования работ на продуктивной сети компании "Дубалайн"

## 1. Общие положения

### 1.1. Назначение документа
Настоящий регламент определяет порядок согласования работ на продуктивной сетевой инфраструктуре компании "Дубалайн" для минимизации рисков нарушения доступности сервисов и обеспечения прозрачности процесса изменения.

### 1.2. Область применения
Регламент распространяется на все работы на продуктивной сети:
- Изменение конфигурации активного сетевого оборудования
- Замена сетевого оборудования
- Модернизация сетевой инфраструктуры
- Проведение профилактических работ
- Исправление инцидентов

### 1.3. Участники процесса

| Роль | Ответственность | Должность |
|------|-----------------|-----------|
| Инициатор | Подача заявки на работы | Любой сотрудник отдела сетевой инфраструктуры |
| Владелец сервиса | Оценка влияния на сервис | Руководитель соответствующего направления |
| Ведущий сетевой инженер (Core Network) | Техническая оценка работ | Ведущий инженер магистральной сети |
| Сетевой инженер (Data Center) | Оценка работ в ЦОД | Инженер ЦОД |
| Сетевой инженер (Campus Network) | Оценка работ в офисной сети | Инженер офисной сети |
| Инженер по эксплуатации (NOC) | Мониторинг и контроль | Инженер NOC |
| Инженер по безопасности (Network Security) | Оценка рисков безопасности | Инженер сетевой безопасности |
| Руководитель отдела сетевой инфраструктуры | Финальное согласование | Руководитель отдела |
| Технический директор | Согласование критических работ | CTO |

## 2. Классификация работ

### 2.1. По уровню критичности

#### Критические работы (Категория 1)
Работы с высоким риском массового сбоя:
- Замена магистрального маршрутизатора
- Изменение схемы маршрутизации в Core Network
- Модернизация core коммутаторов
- Изменение конфигурации файрволов на периметре

**Требуют согласования**: Руководитель отдела + CTO + Владелец сервиса

#### Значительные работы (Категория 2)
Работы с потенциальным влиянием на критичные сервисы:
- Замена коммутаторов в ЦОД
- Изменение конфигурации VLAN в офисной сети
- Модернизация access коммутаторов
- Обновление прошивки сетевого оборудования

**Требуют согласования**: Руководитель отдела + Владелец сервиса

#### Стандартные работы (Категория 3)
Предопределенные работы с известными рисками:
- Добавление нового порта на коммутаторе
- Изменение описания интерфейса
- Настройка нового access коммутатора
- Рутинное обновление прошивки (minor version)

**Требуют согласования**: Ведущий инженер направления

### 2.2. По типу воздействия

| Тип | Описание | Примеры |
|-----|----------|---------|
| Изолированные | Влияние только на одно устройство | Замена одного коммутатора access |
| Локальные | Влияние на один сегмент сети | Работы в одном VLAN |
| Региональные | Влияние на один контур | Работы в одном ЦОД |
| Глобальные | Влияние на всю сеть | Работы в Core Network |

## 3. Процесс согласования

### 3.1. Шаг 1: Инициация заявки

**Действия инициатора:**
1. Создание заявки в системе управления изменениями
2. Описание работ с указанием:
   - Цель работ
   - Оборудование/сервисы, затронутые работами
   - Ожидаемое время простоя
   - План отката
   - Категория работ
3. Прикрепление технической документации

**Срок**: не позднее чем за 5 рабочих дней до работ для категорий 1-2, за 2 дня для категории 3

### 3.2. Шаг 2: Техническая оценка

**Ответственный**: Ведущий сетевой инженер соответствующего направления

**Действия:**
1. Проверка технической выполнимости работ
2. Оценка влияния на сеть
3. Проверка наличия необходимых ресурсов
4. Определение окна обслуживания
5. Подготовка детального плана работ

**Результат**: Техническое заключение (Одобрено/Отклонено/Требует доработки)

### 3.3. Шаг 3: Оценка рисков безопасности

**Ответственный**: Инженер по безопасности (Network Security)

**Действия:**
1. Проверка влияния на политик безопасности
2. Оценка рисков компрометации
3. Проверка соответствия политикам компании
4. Определение требований к мониторингу

**Результат**: Заключение по безопасности (Одобрено/Отклонено/Требует доработки)

### 3.4. Шаг 4: Оценка влияния на сервисы

**Ответственный**: Владелец сервиса

**Действия:**
1. Определение затронутых сервисов
2. Оценка влияния на пользователей
3. Согласование окна обслуживания
4. Определение требований к коммуникации

**Результат**: Согласование владельца сервиса

### 3.5. Шаг 5: Финальное согласование

**Категория 1 (Критические):**
- Руководитель отдела сетевой инфраструктуры
- Владелец сервиса
- Технический директор (CTO)

**Категория 2 (Значительные):**
- Руководитель отдела сетевой инфраструктуры
- Владелец сервиса

**Категория 3 (Стандартные):**
- Ведущий инженер направления

### 3.6. Шаг 6: Уведомление заинтересованных сторон

**За 3 дня до работ (категории 1-2):**
- Уведомление всех владельцев затронутых сервисов
- Уведомление службы поддержки (Help Desk)
- Уведомление NOC

**За 1 день до работ:**
- Финальное подтверждение окна обслуживания
- Уведомление о начале работ

## 4. Критерии отклонения заявки

Заявка может быть отклонена в следующих случаях:
1. Недостаточное техническое обоснование
2. Отсутствие плана отката
3. Несогласованное окно обслуживания
4. Несоответствие политикам безопасности
5. Отсутствие необходимых ресурсов
6. Наличие критических инцидентов в сети
7. Период повышенной нагрузки (например, конец квартала)

## 5. Особенности экстренных работ

### 5.1. Основания для экстренного согласования
- Критический инцидент, требующий немедленного вмешательства
- Угроза безопасности данных
- Критическое нарушение SLA

### 5.2. Процесс экстренного согласования
1. Инициатор создает заявку с пометкой "Экстренно"
2. Техническая оценка выполняется в течение 1 часа
3. Согласование упрощено до:
   - Руководитель отдела сетевой инфраструктуры
   - Владелец сервиса (если доступен)
4. Уведомление заинтересованных сторон после начала работ
5. Пост-анализ в течение 24 часов после работ

## 6. Документирование

### 6.1. Обязательные документы
- Заявка на работы
- Техническое заключение
- План работ
- План отката
- Отчет о выполнении работ

### 6.2. Хранение документов
Все документы хранятся в системе управления изменениями в течение 3 лет.

## 7. Контроль и аудит

### 7.1. Контроль выполнения
- NOC мониторит процесс работ в реальном времени
- Владелец сервиса подтверждает успешное завершение работ
- Инициатор заполняет отчет о выполнении работ

### 7.2. Аудит процесса
Квартальный аудит процесса согласования работ:
- Проверка соблюдения регламента
- Анализ отклонений
- Определение улучшений процесса
