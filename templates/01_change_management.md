# Регламент управления изменениями в сетевой инфраструктуре

## 1. Общие положения

### 1.1. Назначение документа
Настоящий регламент определяет порядок планирования, согласования, выполнения и контроля изменений в сетевой инфраструктуре компании "Дубалайн" с целью минимизации рисков нарушения доступности сервисов и обеспечения стабильной работы IT-инфраструктуры.

### 1.2. Область применения
Регламент распространяется на все изменения в сетевой инфраструктуре:
- Активное сетевое оборудование (коммутаторы, маршрутизаторы, файрволы)
- Сетевые сервисы (DHCP, DNS, NTP, мониторинг)
- Конфигурации VLAN, маршрутизации, политик безопасности
- Физические изменения (перекоммутация, прокладка кабелей)
- Изменения в системах управления сетью

### 1.3. Термины и определения

| Термин | Определение |
|--------|-------------|
| **Изменение** | Любое изменение в конфигурации, топологии или параметрах сетевой инфраструктуры |
| **Стандартное изменение** | Предопределенное изменение с известными рисками и процедурой выполнения |
| **Значительное изменение** | Изменение с потенциальным влиянием на критичные сервисы |
| **Экстренное изменение** | Неотложное изменение для устранения критического инцидента |
| **Окно обслуживания** | Запланированный период времени для выполнения изменений |
| **План отката** | Детальные шаги для возвращения к исходному состоянию |
| **Кабинет изменений** | Ежегодное совещание для планирования изменений |
| **Владелец сервиса** | Ответственное лицо за конкретный IT-сервис |
| **Технический специалист** | Исполнитель технических работ по изменению |

### 1.4. Классификация изменений

#### 1.4.1. По уровню риска
- **Низкий риск**: Стандартные изменения с проверенной процедурой
- **Средний риск**: Значительные изменения с минимальным влиянием на сервисы
- **Высокий риск**: Критические изменения с потенциальным простоем сервисов
- **Критический риск**: Изменения с высоким риском массового сбоя

#### 1.4.2. По типу воздействия
- **Простые**: Изменение одного параметра/устройства
- **Сложные**: Множественные изменения в разных компонентах
- **Комплексные**: Изменения затрагивающие несколько систем/сервисов

## 2. Процесс управления изменениями

### 2.1. Инициация изменения

#### 2.1.1. Основания для инициации
- Запрос на изменение от владельца сервиса
- Результат анализа инцидента
- План технического развития
- Требования безопасности
- Обновление оборудования/ПО

#### 2.1.2. Форма запроса на изменение
Запрос должен содержать:
- Наименование и краткое описание изменения
- Обоснование необходимости (бизнес-потребность)
- Владельца изменения и ответственных исполнителей
- Предполагаемые сроки выполнения
- Оценку влияния на сервисы
- Предварительную оценку рисков
- Требуемые ресурсы

### 2.2. Анализ и оценка изменения

#### 2.2.1. Технический анализ
- Проверка технической реализуемости
- Анализ зависимости от других систем
- Оценка требуемых ресурсов
- Определение последовательности действий
- Выявление потенциальных проблем

#### 2.2.2. Оценка рисков
Матрица оценки рисков:

| Вероятность/Влияние | Низкое | Среднее | Высокое | Критическое |
|-------------------|--------|---------|---------|-------------|
| **Низкая** | Низкий | Низкий | Средний | Средний |
| **Средняя** | Низкий | Средний | Средний | Высокий |
| **Высокая** | Средний | Средний | Высокий | Критический |
| **Очень высокая** | Средний | Высокий | Критический | Критический |

#### 2.2.3. Оценка воздействия на сервисы
- Критичные сервисы (Core Network, Internet, VPN)
- Важные сервисы (Internal Services, Monitoring)
- Второстепенные сервисы (Test/Dev environments)

### 2.3. Планирование изменения

#### 2.3.1. Разработка плана выполнения
План должен включать:
- Детальную последовательность действий
- Временные рамки для каждого шага
- Точки контроля и проверки
- Ответственных за каждый этап
- Требуемые доступы и инструменты

#### 2.3.2. План отката
Обязательные элементы плана отката:
- Условия для инициации отката
- Последовательность шагов отката
- Максимальное время отката
- Проверка успешности отката
- Уведомление заинтересованных лиц

#### 2.3.3. План тестирования
- Методы тестирования до внедрения
- Критерии успешности тестов
- План тестирования в production среде
- Процедуры отката при неудачном тесте

### 2.4. Согласование изменения

#### 2.4.1. Уровни согласования
**Уровень 1 (Низкий риск)**:
- Согласование с руководителем отдела сетевой инфраструктуры
- Уведомление владельцев затрагиваемых сервисов

**Уровень 2 (Средний риск)**:
- Согласование с руководителем отдела ИТ
- Согласование с руководителем отдела информационной безопасности
- Формальное согласование владельцев сервисов

**Уровень 3 (Высокий/Критический риск)**:
- Согласование с CIO
- Согласование с руководителем отдела информационной безопасности
- Согласование с руководителями бизнес-подразделений
- Утверждение на Кабинете изменений

#### 2.4.2. Процедура согласования
1. Рассылка уведомления о запланированном изменении
2. Сбор отзывов и комментариев (минимум 3 рабочих дня)
3. Проведение технического совещания (при необходимости)
4. Фиксация результатов согласования
5. Получение финального утверждения

### 2.5. Подготовка к выполнению

#### 2.5.1. Техническая подготовка
- Подготовка оборудования и ПО
- Создание резервных копий конфигураций
- Подготовка инструментов мониторинга
- Проверка доступов и прав

#### 2.5.2. Коммуникационная подготовка
- Формирование списка уведомляемых лиц
- Подготовка шаблонов уведомлений
- Определение каналов коммуникации
- Назначение ответственного за коммуникации

#### 2.5.3. Подготовка команды
- Распределение ролей и обязанностей
- Проведение технического брифинга
- Подготовка контактной информации
- Определение эскалационного пути

## 3. Выполнение изменения

### 3.1. Предварительные проверки

#### 3.1.1. Проверка готовности систем
- Статус мониторинга (все системы в норме)
- Доступность резервных систем
- Готовность команды исполнителей
- Доступность коммуникационных каналов

#### 3.1.2. Базовые метрики
- Загрузка сетевых устройств
- Доступность ключевых сервисов
- Время отклика критичных приложений
- Статус систем безопасности

### 3.2. Процедура выполнения

#### 3.2.1. Начало окна обслуживания
```
Время: T-30 минут
Действия:
1. Проверка статуса всех систем
2. Уведомление о начале окна обслуживания
3. Активация дополнительного мониторинга
4. Финальная проверка готовности команды
```

#### 3.2.2. Выполнение технических работ
```
Время: T-15 минут до T+X минут
Действия:
1. Создание резервной копии текущей конфигурации
2. Выполнение изменений согласно плану
3. Проверка каждого шага
4. Фиксация результатов
5. Мониторинг состояния систем
```

#### 3.2.3. Тестирование результатов
```
Время: T+X минут до T+X+30 минут
Действия:
1. Базовая проверка доступности
2. Функциональное тестирование
3. Проверка производительности
4. Тестирование отказоустойчивости
5. Сбор метрик производительности
```

### 3.3. Мониторинг в процессе выполнения

#### 3.3.1. Ключевые метрики
- Доступность сетевых устройств (ping, SNMP)
- Загрузка процессора и памяти
- Utilization интерфейсов
- Количество ошибок на интерфейсах
- Время отклика сервисов

#### 3.3.2. Пороговые значения
| Метрика | Норма | Предупреждение | Критично |
|---------|-------|----------------|----------|
| CPU Utilization | < 70% | 70-85% | > 85% |
| Memory Utilization | < 80% | 80-90% | > 90% |
| Interface Errors | < 0.1% | 0.1-1% | > 1% |
| Response Time | < 100ms | 100-500ms | > 500ms |

### 3.4. Процедура отката

#### 3.4.1. Условия для отката
- Превышение пороговых значений метрик
- Недоступность критичных сервисов
- Обнаружение непредвиденных побочных эффектов
- Превышение времени выполнения изменения

#### 3.4.2. Процесс отката
1. Немедленное уведомление руководителя изменения
2. Инициация процедуры отката согласно плану
3. Мониторинг процесса отката
4. Проверка успешности отката
5. Анализ причин неудачи
6. Формирование отчета об инциденте

## 4. Завершение изменения

### 4.1. Пост-реализационное тестирование

#### 4.1.1. Функциональное тестирование
- Проверка всех затронутых сервисов
- Тестирование пользовательских сценариев
- Проверка резервных каналов
- Тестирование систем безопасности

#### 4.1.2. Нагрузочное тестирование
- Симуляция пиковых нагрузок
- Тестирование отказоустойчивости
- Проверка масштабируемости
- Мониторинг производительности

### 4.2. Документирование

#### 4.2.1. Техническая документация
- Актуализация сетевых диаграмм
- Обновление конфигурационной документации
- Фиксация изменений в CMDB
- Обновление эксплуатационной документации

#### 4.2.2. Отчет о выполнении
Отчет должен содержать:
- Фактическое время выполнения
- Результаты тестирования
- Обнаруженные проблемы и их решение
- Уроки извлеченные из опыта
- Рекомендации по будущим изменениям

### 4.3. Закрытие изменения

#### 4.3.1. Критерии успешного закрытия
- Все тесты пройдены успешно
- Документация обновлена
- Заинтересованные стороны уведомлены
- Нет открытых инцидентов связанных с изменением

#### 4.3.2. Финальное уведомление
- Уведомление о завершении работ
- Отчет о результатах выполнения
- Информация о доступности сервисов
- Контактная информация для экстренной связи

## 5. Управление экстренными изменениями

### 5.1. Критерии экстренного изменения
- Критический инцидент с массовым влиянием
- Угроза безопасности данных
- Недоступность критичных бизнес-процессов
- Требования регуляторов

### 5.2. Процедура экстренного изменения

#### 5.2.1. Инициация
1. Обнаружение критической проблемы
2. Оценка срочности и влияния
3. Принятие решения об экстренном изменении
4. Формирование экстренной команды

#### 5.2.2. Выполнение
1. Минимальное необходимое изменение
2. Постоянный мониторинг
3. Готовность к немедленному откату
4. Фиксация всех действий

#### 5.2.3. Пост-экстренные действия
1. Анализ причин инцидента
2. Планирование постоянного решения
3. Обновление документации
4. Извлечение уроков

## 6. Метрики и KPI

### 6.1. KPI процесса управления изменениями
- Процент успешных изменений (>95%)
- Среднее время выполнения изменения
- Количество откатов за период (<5%)
- Время согласования изменений
- Удовлетворенность заказчиков

### 6.2. Отчетность
- Еженедельный отчет о выполненных изменениях
- Ежемесячный анализ KPI
- Квартальный обзор процесса
- Годовой отчет об эффективности

## 7. Роли и ответственности

### 7.1. Руководитель изменения
- Общая ответственность за изменение
- Координация работы команды
- Управление рисками
- Коммуникация с заинтересованными сторонами

### 7.2. Технический специалист
- Выполнение технических работ
- Подготовка технической документации
- Тестирование результатов
- Участие в анализе проблем

### 7.3. Владелец сервиса
- Оценка бизнес-влияния изменения
- Согласование времени выполнения
- Участие в тестировании
- Приемка результатов

### 7.4. Менеджер по безопасности
- Оценка рисков безопасности
- Проверка соответствия политикам
- Участие в тестировании безопасности
- Мониторинг после внедрения

### 8.2. Чек-лист проверки готовности

```
□ Резервные копии созданы и проверены
□ Доступы к системам получены и проверены
□ Команда уведомлена о времени начала
□ Мониторинг активирован
□ Контакты экстренной связи актуальны
□ План отката доступен и понятен
□ Необходимые инструменты подготовлены
□ Владельцы сервисов уведомлены
```

### 8.4. Пример конфигурации коммутатора после изменения

```cisco
! Catalyst 9500 - Core Switch
version 17.3.1
hostname CORE-SW-01
!
interface TenGigabitEthernet1/1/1
 description Uplink to ISP-01
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30,40,50
 spanning-tree portfast trunk
!
interface Vlan10
 description Management Network
 ip address 10.1.10.1 255.255.255.0
!
interface Vlan50
 description New Guest Network
 ip address 10.1.50.1 255.255.255.0
 ip access-list GUEST_ACL
!
router ospf 1
 router-id 10.1.10.1
 network 10.1.10.0 0.0.0.255 area 0
 network 10.1.50.0 0.0.0.255 area 0
!
ip access-list extended GUEST_ACL
 permit udp any any eq domain
 permit udp any any eq bootpc
 permit tcp any any established
 deny ip any any
!
line vty 0 4
 login local
 transport input ssh
!
```
